Building a "Meta-Synced" Block using useEntityProp

A very common annoyance is deciding where to store block data. Most blocks save data inside the post content using HTML comments.

But what if you need that data to be searchable, filterable, or used by another plugin? Then you should use Post-Meta.

The following block creates a custom metafield attached to the post itself. Use this method for "Review Scores," "Price Tags," or "Sponsored Post" toggles.

1. The PHP (Essential for REST API access)

You must register the meta field in your PHP. Otherwise, the Block Editor's request will be blocked for security reasons.

File: plugin.php 
add_action( 'init', function() {
    register_post_meta( 'post', '_my_custom_rating', [
        'show_in_rest' => true,
        'single'       => true,
        'type'         => 'number',
    ] );
} );

2. The Editor Logic

File: edit.js 
import { useEntityProp } from '@wordpress/core-data';
import { useBlockProps } from '@wordpress/block-editor';
import { RangeControl } from '@wordpress/components';

export default function Edit( { context } ) {
    const blockProps = useBlockProps();
    const postType = context.postType;

    // This hook acts like useState, but targets database meta!
    const [ meta, setMeta ] = useEntityProp( 'postType', postType, 'meta' );

    const rating = meta[ '_my_custom_rating' ] || 0;

    const updateRating = ( newValue ) => {
        setMeta( { ...meta, _my_custom_rating: newValue } );
    };

    return (
        <div { ...blockProps }>
            <RangeControl
                label="Rating Score"
                value={ rating }
                onChange={ updateRating }
                min={0} max={5} step={0.5}
            />
            <p>Saved to <strong>wp_postmeta</strong>, not post_content.</p>
        </div>
    );
}

3. The Configuration

Crucially, you have to tell the block to "listen" to the post type context so it knows which post it belongs to.

File: block.json 
{
    "usesContext": [ "postType" ]
}
